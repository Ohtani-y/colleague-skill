# Work Skill 分析プロンプト

## タスク

**{name}** の原材料（ドキュメント、メッセージ、メールなど）を受け取ります。
そこから仕事の能力と手法を抽出し、Work Skillの構築に使用します。

**原則：仕事に関連する内容のみ抽出し、雑談は無視する。推測はせず、根拠がある場合のみ記載し、なければ「原材料不足」と注記する。**

---

## 共通抽出項目（全職種共通）

### 1. 担当範囲

原材料から識別する：
- 担当しているシステム／モジュール／事業ライン／プロダクト
- 管理しているドキュメント（API仕様書、wiki、runbookなど）
- 職責の境界（どこまでが自分の範囲で、どこからが範囲外か）
- 頻繁に言及するプロジェクトコード名、業務用語

```
出力形式：
担当領域：[記述]
中核システム：[リスト]
管理ドキュメント：[リスト]
境界：[担当範囲／担当外]
```

### 2. 業務フロー

タスク記述や会議議事録から抽出する：
- タスクを受けた際の処理ステップ
- 企画書／ドキュメント作成時の構成パターン
- 進捗管理とデッドライン対応の方法
- 異常時／緊急時の対応方法

```
出力形式：
タスク受領時：[ステップ]
企画書作成：[構成の記述]
異常対応：[フロー]
```

### 3. アウトプット形式の傾向

- 表／リスト／フローチャート／プレーンテキストの使い分け
- 結論先行型か、順を追って説明する型か
- ドキュメントの詳細度（ミニマル／標準／詳細）
- 返信／メールのスタイル

```
出力形式：
ドキュメントスタイル：[記述]
詳細度：[ミニマル／標準／詳細]
```

### 4. 経験・ナレッジベース

本人が明確に述べた経験則、踏んだ地雷、技術的見解（原文を直接引用）：

```
- "[原文または要約]"
- "[原文または要約]"
```

## 職種別の専門抽出項目

{name}の職種に応じて、該当する項目を重点的に抽出する：

---

### 🖥️ バックエンドエンジニア / サーバーサイドエンジニア

**技術規約**：
- 技術スタック（言語、フレームワーク、ミドルウェア）
- 命名規則（APIパスのスタイル、変数／関数の命名）
- API設計（レスポンス構造、エラーコード、ページネーション、冪等性）
- DB操作の傾向（ORM vs 生SQL、トランザクション境界）
- 例外処理のスタイル

**Code Reviewの重点**：
- 繰り返し指摘するCRの問題（N+1、トランザクション、並行処理の安全性など）
- CRコメントのスタイル（直接的／婉曲、[block]/[suggest]の使い分けなど）

**デプロイ・運用**：
- 重視するモニタリング指標
- 本番障害の調査手順
- 変更リリースのフロー

---

### 🌐 フロントエンドエンジニア

**技術規約**：
- 技術スタック（フレームワーク、状態管理、スタイリング方式）
- コンポーネント分割の方針（いつ分割し、いつ分割しないか）
- パフォーマンスの関心事（初期表示、遅延読み込み、バンドルサイズなど）
- API呼び出しとエラーハンドリングの方法

**エンジニアリングプラクティス**：
- コード規約ツール（ESLintルール、Prettierの設定傾向）
- テストカバレッジの要件（ユニットテスト／統合テストへの姿勢）
- CRの重点（アクセシビリティ／レスポンシブ／互換性への関心度）

---

### 🤖 アルゴリズムエンジニア / MLエンジニア

**研究と実験**：
- 問題定義の方法（ML問題をどう分解するか）
- 実験設計のパターン（ベースライン選択、ablation設計）
- 指標定義の傾向（オフライン指標 vs オンライン指標への姿勢）
- 常用するモデル／手法

**エンジニアリング実装**：
- 学習フレームワークの選好
- モデルデプロイのフロー
- データ前処理の規約

**ドキュメントと結論**：
- 実験レポートの書き方（結論重視／過程重視）
- 引用するpaper・手法

---

### 📱 プロダクトマネージャー / テクニカルPM

**要件処理**：
- PRDの構成と詳細度
- ユーザーストーリー／要件境界の定義方法
- 開発チームとの認識合わせ方法（レビュー方式、修正フロー）

**意思決定フレームワーク**：
- 優先順位付けの方法（RICE/MoSCoW/独自）
- データドリブン vs 直感の比率
- 要件の衝突をどう処理するか

**成果物**：
- 納品するドキュメントの種類（PRD/MRD/プロトタイプ/競合分析）
- プロトタイピングツールの選好
- データ計測（イベントトラッキング）への関与度

---

### 🎨 デザイナー

**デザイン規約**：
- 使用するデザインシステム／コンポーネントライブラリ
- 仕様の注記方法と納品規約
- ピクセルパーフェクトへのこだわり度

**業務フロー**：
- 要件からデザイン案までのステップ
- デザインレビュー／検収の方法
- 開発側の再現度に関する問題への対処

---

### 📊 データアナリスト

**分析手法**：
- 常用する分析フレームワーク（ファネル／コホート／A/Bテストなど）
- SQLスタイル（簡潔型／コメント詳細型）
- データ可視化の傾向（グラフの種類選択）

**レポートスタイル**：
- 結論 vs データの比率
- 「データで語る」の実践度
- データ異常や定義の相違への対処方法

---

## 出力要件

- 言語：日本語
- 情報がない項目は：`（原材料不足、関連ドキュメントの追加を推奨）` と注記
- 原文に根拠がある結論は：引用符で原文を記載
- 出力結果はwork.mdの生成に直接使用するため、具体的かつ実行可能な記述にすること。「おそらく」「傾向がある」などの曖昧な表現は使わない
