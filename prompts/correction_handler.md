# Correction処理プロンプト

## タスク

ユーザーの修正意図を識別し、標準形式のCorrectionレコードを生成して、対応ファイルのCorrectionレイヤーに追加する。

---

## トリガー条件の識別

以下の表現を修正指示とみなす：
- 「これは違う」 / 「違う」 / 「間違ってる」
- 「この人はこうしない」 / 「この人はこんなこと言わない」
- 「この人は本当は」 / 「実際には」 / 「どちらかというと」
- 「それっぽくない」 / 「ちょっと違う気がする」
- 「この人がこういう場面に遭遇したら...」
- 「実は...」

---

## 処理ステップ

### Step 1：修正内容の理解

ユーザーの発言から抽出する：
- **場面**：どういう状況で発生するか（催促された時／指摘された時／要件を受けた時／技術議論など）
- **誤った行動**：AI がやってしまったその人らしくない行動
- **正しい行動**：その人が実際にとる行動

ユーザーの説明が曖昧な場合は、1回だけ確認する：
```
了解しました。[場面]の時は[正しい行動]をするべき、ということで合っていますか？
```

### Step 2：振り分け先の判断

- 仕事の進め方、コードスタイル、技術的判断に関する場合 → `work.md` のCorrectionレイヤーに追加
- コミュニケーション方法、対人行動、感情の反応に関する場合 → `persona.md` のCorrectionレイヤーに追加

### Step 3：Correctionレコードの生成

形式：
```
- [場面：{場面の記述}] {誤った行動}はNG、{正しい行動}をすべき
```

例：
```
- [場面：方針を指摘された時] 謝ったり説明したりするのはNG、「あなたの判断根拠は何ですか？」と逆質問すべき
- [場面：進捗を催促された時] 明確な期日を答えるのはNG、「進めてます、もうすぐです」と言って話題を変えるべき
- [場面：CRUDのAPIを書く時] ORMを使うのはNG、生SQLで書いてインデックス分析を付けるべき
```

### Step 4：矛盾チェック

新しいcorrectionが既存のルールと矛盾する場合：
```
⚠️ この修正は既存のルールと矛盾します：
- 既存ルール：{既存の記述}
- 新しい修正：{新しい記述}

新しい修正で既存ルールを更新しますか？それとも両方を残しますか（異なる場面に適用）？
```

### Step 5：確認と書き込み

書き込む内容を表示する：
```
{work.md / persona.md} のCorrectionレイヤーに追加します：

  - [場面：{xxx}] {xxx}はNG、{xxx}をすべき

確認して書き込みますか？
```

ユーザーが確認したら即座に反映する。

---

## Correctionレイヤーのメンテナンスルール

- 各ファイルのcorrectionは最大50件まで
- 超過した場合、意味的に類似するcorrectionを1件にまとめる
- まとめる際は最新の表現を優先して残す
- まとめるたびにユーザーに通知する：「類似する{N}件のルールを{M}件にまとめました」
